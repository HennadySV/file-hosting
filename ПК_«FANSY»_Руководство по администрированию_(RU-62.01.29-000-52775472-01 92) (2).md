# Введение

Настоящее руководство регламентирует порядок администрирования программного комплекса «FANSY» (далее — ПК «FANSY»), включающего следующие приложения:

1. Система внутреннего учета «FANSY-SPECTRE».
2. Система внутреннего учета депозитария «FANSY-DEPO».
3. Универсальная учетная система НФО «FANSY-STANDARD».
4. Система автоматизации бэк-офиса фондового отдела банка «FANSY-TRUST».
5. Система внутреннего учета специализированного регистратора «FANSY-REESTR».

Администратором является пользователь с идентификатором, содержащим префикс «ADMIN». Администратору предоставлен доступ к форме «Справочник операционистов», предназначенной для управления учетными записями пользователей и назначения прав доступа к формам и документам приложений.

# Уровни защиты данных

ПК «FANSY» обеспечивает три уровня защиты данных от несанкционированного доступа:

1. **Физический уровень**: права доступа, определяемые на уровне СУБД.
2. **Логический уровень**: правовой профиль пользователя, включающий префикс идентификатора, перечень рабочих мест, ограничения на данные и флаг доступа к персональным данным. Проверка выполняется при работе с приложениями.

Логический уровень включает два подуровня:

* **Профиль прав**: определяет доступ к формам, меню и кнопкам на основе префикса идентификатора пользователя.
* **Профиль ограничений**: задается в форме «Справочник операционистов» и ограничивает действия (вставка, изменение, удаление, просмотр, фильтрация, импорт, экспорт) на формах, доступных пользователю.

Права доступа зависят от префикса идентификатора:

* **SYSDBA**: неограниченные права на все объекты и действия и используется только при первоначальной настройке Системы, а, также, для заведения в систему рабочего места «ADMIN».
* **ADMIN**: максимальные права, включая просмотр и редактирование всех форм, кроме документов.
* **USER**: стандартные права, включающие просмотр всех форм (кроме административных) и редактирование разрешенных форм.
* **GUEST**: минимальные права, ограничивающиеся просмотром отдельных форм без возможности редактирования.

# Режимы работы ПК «FANSY»

ПК «FANSY» имеет единственный режим работы — штатный.

Штатный режим предусматривает запуск клиентского приложения, которое устанавливает соединение с СУБД, расположенной на сервере, и обеспечивает доступ к базам данных ПК «FANSY».

При инициализации клиентское приложение подключается к штатному порту СУБД:

* **tcp/3050** — для РЭД База данных;
* **tcp/5432** — для PostgreSQL.

При поступлении запроса на подключение сервер СУБД выполняет проверку учётных данных и прав доступа. При успешной аутентификации и авторизации устанавливается соединение, после чего начинается обработка запросов клиента.

В штатном режиме ПК «FANSY» использует следующие файлы:

* файлы баз данных;
* файлы шаблонов отчётов;
* файлы журналов.

Файлы журналов предназначены для хранения параметров настроек приложения и обеспечения его функционирования.

Штатный режим работы ПК «FANSY» реализует все предусмотренные механизмы защиты информации, включая:

* идентификацию и аутентификацию;
* разграничение доступа;
* регистрацию событий безопасности;

# Принципы реализации функций безопасности

ПК «FANSY» реализует функции безопасности на основе принципа минимально необходимых прав, атрибутивного управления доступом и централизованного администрирования через справочник операционистов и базу метаданных (bal\_meta из состава дистрибутива). Все механизмы защиты информации соответствуют требованиям мер ИАФ.3, ИАФ.5, УПД.1 (усиление 1), УПД.2, РСБ.1, РСБ.2, изложенным в п. 2.5 Технических условий (Программный комплекс «FANSY» «Технические условия (ТУ 62.01.29-000-52775472-2024)»).

## Идентификация и аутентификация

Принцип: каждый субъект доступа однозначно идентифицируется и аутентифицируется до получения доступа к ресурсам системы.

1. В ПК FANSY обеспечивается формирование идентификатора (логина), который однозначно идентифицирует пользователя. Формирование идентификатора должно обеспечиваться заданием его уникального значения при создании нового пользователя администратором ПК FANSY. В ПК FANSY осуществляется присвоение идентификатора (логина) пользователю, выполняемое при создании пользователя с заданными значениями идентификатора и других атрибутов профиля пользователя.

Реализация:

Формирование идентификатора (логина) осуществляется администратором при создании учётной записи. Логин должен начинаться с одного из префиксов[[1]](#footnote-1):

* + ADMIN\_ — административные права;
  + USER\_ — права обычного пользователя;
  + GUEST\_ — ограниченные (гостевые) права.

Длина логина — не более 24 символов. Уникальность обеспечивается на уровне СУБД.

В ПК FANSY осуществляется защита аутентификационной информации в процессе ее ввода для аутентификации от возможного использования лицами, не имеющими на это полномочий. Защита обратной связи «система – субъект доступа» в процессе аутентификации обеспечивается исключением отображения для пользователя действительного значения аутентификационной информации и (или) количества вводимых пользователем символов аутентификационной информации. Вводимые символы пароля могут отображаться условными знаками «\*», «•» или иными знаками.

Реализация:

Аутентификация выполняется по паролю, хранимому в хэшированном виде в таблице пользователей СУБД. При вводе пароля обратная связь маскируется символами •, исключая отображение реального значения и количества символов~~.~~

1. В ПК FANSY В ПК FANSY обеспечивается возможность блокирования идентификатора пользователя через период времени неиспользования – 90 дней (период заданный администратором).

Реализация:

Учётная запись автоматически блокируется по истечении заданного количества дней неактивности (по умолчанию — 90 дней). Параметр настраивается в справочнике операционистов (пункт 6.3 Руководства по установке и настройке (RU-62.01.29-000-52775472-01 85)) (в соответствии с УПД.1, усиление 1).

## Управление учётными записями

Принцип: Управление идентификаторами, в том числе создание, присвоение, уничтожение идентификаторов, в том числе внешних пользователей .

Реализация:

1. Все операции с учётными записями выполняются через СУБД и справочник операционистов с учетной записью с правами администратора в интерфейсе ПК «FANSY».
2. Создание учётной записи включает (см. раздел 6 даного руководства):
   * заведение учетной записи с необходимым префиксом в СУБД и установка пароля для данной записи;
   * Создание записи в справочнике операционистов с логином, ранее заведенным в СУБД в качестве идентификатора операциониста;
   * Присвоение необходимых рабочих мест для завденного, в справочнике операционистов пользователя, согласно штатному расписанию );
   * назначение профиля ограничений (при необходимости).
3. При необходимости блокирование учетной записи осущетствляется выставлением даты блокировки в справочнике операционистов на закладке «Сведения». Также там настраивается автоматическая блокировка записи по истечению опредленного количества дней не активности учтенной записи (см. раздел 8 данного руковдства)
   * Все заведенные в справочнике учетнные записи, по умолчанию имеют неограниченный срок действия (с даты заведения в систему по 01.01.9999), но при необходимости (на пример для гостевых записей) можно изменить срок действия записи путем выставления, обусловленной необходимостью, даты завершения
4. Журналирование изменений (см. п. 11 данного руководства):
   * изменение ограничений;
   * смена рабочих мест;
   * изменение роли или профиля;
   * изменение личных данных.

## Разграничение доступа

Принцип: атрибутивное управление доступом, предусматривает управление доступом субъектов доступа (пользователей ПК FANSY) к наименованным объектам (справочники, выборки данных) при выполнении операций чтение, запись, модификация.

Реализация:

1. Субъекты доступа характеризуются:
   * префиксом логина (ADMIN\_, USER\_, GUEST\_) — определяет базовый набор прав (чтение/запись/изменение);
   * перечнем рабочих мест[[2]](#footnote-2) — ограничивает доступ с конкретных станций;
   * профилем ограничений — дополнительные атрибуты, уточняющие доступ к справочникам и формам.
2. Объекты доступа (справочники, документы, выборки данных) характеризуются:
   * идентификатором объекта (для форм и справочников);
   * категорией и состоянием (для документов).
3. Правила разграничения хранятся в метаданных, СУБД и библиотеке функций. При выполнении операции (чтение, запись, модификация) система проверяет:
   * Соответствие префикса разрешённым;
   * Наличие рабочего места в списке;
   * Наличие права в профиле ограничений;
   * Соответствие категории/состояния документа.

## Регистрация событий безопасности

Принцип: регистрация всех событий, отражающих: вход (выход), а также попытки входа субъектов доступа (пользователей) в ПК FANSY, создание (архивация) пользователей, изменение статуса учетной записи, изменение правил разграничения доступа.

Реализация:

Для просмотра перечня зарегистрированных событий используется форма «Мониторинг транзакций» в интерфейсе ПК «FANSY» (раздел 11 данной документации)

1. Регистрируемые события:
   * вход/выход пользователя (включая неуспешные попытки) Тип действия «CONNECT» и «DISСONNECT» в столбце «Тип действия». Неудачные попытки входа обозначаются красным, тип действия при этом также «CONNECT»;
   * создание, архивация, блокировка, разблокировка учётной записи. Тип действия «UPDATE», при этом на закладке «Ординарные поля» указано символом «\*» измененное поле;
   * изменение правил разграничения доступа. Тип действия «INSERT» на закладке «Ординарные поля» прописывается присвоенное рабочее место или профиль ограничения. Тип действия «DELETE» в случае удаления рабочего места ли профиля ограничения, на закладке «Ординарные поля» прописывается название удаленного рабочего места или профиля ограничений .
2. Состав записи в журнале:
   * тип события;
   * дата и время;
   * идентификатор субъекта (логин);
   * детали изменения (старое значение, новое значение);
   * результат операции (успех/ошибка).
3. Хранение: журнал ведётся в СУБД, доступ к просмотру — только у пользователей с префиксом ADMIN\_. Срок хранения — настраивается администратором СУБД.

# Принципы безопасной работы

При эксплуатации ПК «FANSY» **обязательно соблюдение** следующих принципов безопасной работы, направленных на поддержание заявленного уровня защищённости информации.

## Принципы работы с учётными записями

1. **Каждый пользователь должен использовать только свою учётную запись.** Запрещается передача логина и пароля третьим лицам.
2. **Пароль должен соответствовать требованиям сложности**:
   * минимальная длина — **8 символов**;
   * обязательное использование **строчных, заглавных букв, цифр и спецсимволов**;
   * **запрещено использование** слов из словаря, имён, дат рождения, последовательностей (12345, qwerty).
3. **Смена пароля** — **не реже одного раза в 60 дней**. Для этого необходимо выставлять период актуальности учетной записи не более 60 дней.
4. **Блокировка при неактивности** (90 дней) — **не отключать**.

## Принципы разграничения доступа

1. **Назначение прав — строго по должностным обязанностям.**.
2. **Рабочие места — только с утверждённых станций.** Запрещается добавление в перечень IP-адресов из внешних сетей.
3. **Принцип минимальных прав:** Если пользователю не требуется доступ к определённому типу документов — **не назначать.**

## Принципы работы с журналом аудита

1. **Журнал аудита — обязательный объект мониторинга.** Администратор **с установленной в организации периодичностью (например, ежедневно или один раз в смену)**  проверяет:
   * неуспешные попытки входа;
   * входы с нестандартных рабочих мест;
   * изменения прав доступа;
   * массовые операции с документами.
2. **Выявление аномалий**. При обнаружении подозрительных событий (например, вход в 3:00 ночи от USER\_PETROV) — **немедленная блокировка и расследование инцидента безопасности.**
3. **Хранение журнала аудита — не менее 6 месяцев**. Экспорт в зашифрованный архив при необходимости.

## Принципы обработки инцидентов

1. **Любая ошибка аутентификации, сбой доступа или подозрительное поведение системы — сообщать администратору немедленно.**

**К ошибкам аутентификации относятся большое (больше трех) количество строк «CONNECT», выделенных красным в Мониторинге транзакций.**

**Сбой доступа – это любая ошибка или диалоговое окно, возникающая при нажатии кнопки «Ок» после ввода логина/пароля в окне авторизации, либо сразу при запуске программы.**

**Под подознительным повденеием системы подразумеваетяс появление диалоговых окон в тех местах, где их ранее не было, например при открытии справочников и не несущих в себе явной смысловой нагрузки а транслирующих внутренние сообщения операционной системы. Например, сообщение со смысловой нагрузкой – «Не достаточно памяти для окончания для отображения формы», сообщение без смысловой нагрузки – «Error read memory 00x884523»**

1. **Запрещается:**
   * отключать механизмы безопасности ~~(«чтобы не мешали»)~~;
   * использовать общие учётные записи (пример, GUEST\_123 для всех);
   * игнорировать предупреждения Системы.

## Общие принципы

1. **Физическая безопасность:** Рабочая станция с клиентом ПК «FANSY» **не должна оставаться без присмотра в разблокированном состоянии.**

## Просмотр и анализ журнала аудита

* **Доступ**: только пользователи с префиксом ADMIN\_. Меню: **«Сервис → Журнал аудита»** или прямой SQL-запрос к таблице аудита.
* **Фильтры анализа**:
  + по дате/времени;
  + по пользователю;
  + по типу события;
  + по рабочему месту (IP/MAC).
* **Рекомендуемая периодичность**:
  + **Ежедневно** — проверка неуспешных входов и входов вне рабочего времени.
  + **Еженедельно** — анализ изменений прав доступа.
  + **Ежемесячно** — экспорт и архивация журнала.
* **Пример тревожных событий**:
  + ≥ 3 неуспешных попытки входа за 5 минут → **немедленная блокировка** учётной записи.
  + Вход с неразрешённого IP → **блокировка + уведомление**.
  + Изменение прав USER\_ на ADMIN\_ → **расследование**.

| **Событие** | **Действие администратора** |
| --- | --- |

| **Блокировка по неактивности** | Связаться с пользователем, подтвердить статус, разблокировать только после проверки. |
| **Изменение прав** | Проверить соответствие должностной инструкции. Откат при нарушении. |
| **Массовые операции** | Анализ через выборку в журнале: кто, когда, какие документы. |

## Генерация проводок

Система поддерживает следующие способы генерации проводок:

1. Проводки создаются на текущую дату.

# Описание сетевых интерфейсов и протоколов библиотеки Synapse

## Общие положения

Настоящий раздел регламентирует характеристики сетевых интерфейсов и протоколов, используемых библиотекой Synapse в программном обеспечении, разработанном на языке Free Pascal. Библиотека Synapse обеспечивает реализацию сетевых коммуникаций с использованием блокирующих (синхронных) сокетов и ограниченного неблокирующего режима.

## Сетевые протоколы

Библиотека Synapse поддерживает **IPv4** (протокол Интернета версии 4):

* Предназначен для передачи данных в сетях TCP/IP.
* Используется для всех поддерживаемых протоколов прикладного уровня.
  2.2. **IPv6** (протокол Интернета версии 6):
* Обеспечивает расширенное адресное пространство и совместимость с современными сетями.
* Применяется для всех поддерживаемых протоколов, где это применимо.

## Транспортные протоколы

Библиотека Synapse реализует **TCP** (протокол управления передачей):

* Надежный протокол с установлением соединения, обеспечивающий гарантированную доставку данных.
* Используется для протоколов HTTP, FTP, SMTP, POP3, IMAP4, Telnet, LDAP.
* Реализован через класс TTCPBlockSocket.
  3.2. **UDP** (протокол пользовательских датаграмм):
* Ненадежный протокол без установления соединения, применяемый для высокоскоростной передачи данных.
* Используется для протоколов DNS, Syslog, SNMP, NTP/SNTP, TFTP.
* Реализован через класс TUDPBlockSocket, поддерживает широковещательную и многоадресную передачу (с настройкой TTL для multicast).

## Протоколы прикладного уровня

Библиотека Synapse поддерживает следующие протоколы прикладного уровня, реализованные в соответствующих модулях:

1. **HTTP/HTTPS** (протокол передачи гипертекста / защищенный):

* Используется для веб-запросов и передачи данных.
* Реализован через модуль httpsend (класс THTTPSend).
* HTTPS требует внешних библиотек шифрования (OpenSSL, Cryptlib).

1. **FTP/TFTP** (протокол передачи файлов / тривиальный протокол передачи файлов):

* FTP (TCP) — для надежной передачи файлов, реализован через модуль ftpsend.
* TFTP (UDP) — для упрощенной передачи файлов, реализован через модуль tftpsend.

1. **SMTP, POP3, IMAP4** (протоколы электронной почты):

* Используются для отправки и получения писем (TCP).
* Реализованы через модули smtpsend, pop3send, imap4send.

1. **DNS** (система доменных имен):

* Используется для разрешения доменных имен (UDP, опционально TCP).
* Реализован через модуль dnssend.

1. **SNMP, NTP/SNTP** (протоколы управления сетью и синхронизации времени):

* SNMP (UDP) — для управления сетевыми устройствами, реализован через модуль snmpsend.
* NTP/SNTP (UDP) — для синхронизации времени, реализован через модуль ntpsend.

1. **Syslog** (протокол системного журналирования):

* Используется для передачи логов (UDP).
* Реализован через модуль syslogsend.

1. **LDAP** (протокол доступа к каталогам):

* Используется для работы с каталогами (TCP).
* Реализован через модуль ldapsend.

1. **Telnet** (протокол удаленного доступа):

* Используется для удаленного управления (TCP).
* Реализован через модуль tlntsend (класс TTelnetSend).

1. **SSH** (протокол защищенного удаленного доступа):

* Используется для защищенного удаленного управления (TCP).
* Реализован через модуль tlntsend с поддержкой SSHv2, требует внешних библиотек шифрования.

## Механизмы шифрования

Библиотека Synapse поддерживает защищенные соединения через протоколы **SSL/TLS** и **SSH**.
Поддерживаемые версии SSL/TLS: SSLv2, SSLv3, TLSv1, TLSv1.1, TLSv1.2, TLSv1.3.

Требуются внешние библиотеки шифрования:

* **OpenSSL**: для SSL/TLS (например, в HTTPS, FTPS).
* **Cryptlib**: для SSL/TLS и SSH, поддерживает частные ключи в специфическом формате.
* **LibSSH2**: для SSH, поддерживает аутентификацию с ключами в формате OpenSSH.

Настройка шифрования требует установки библиотек и их библиотечных файлов (например, CL32.DLL для Cryptlib на Windows).

## Сетевые интерфейсы

Сетевые интерфейсы библиотеки Synapse реализованы через следующие классы:
**TTCPBlockSocket**:

Назначение: реализация клиентских и серверных TCP-соединений.

* Поддерживаемые протоколы: IPv4, IPv6.
* Основные функции: подключение к удаленному хосту, отправка и получение данных, настройка таймаутов, прослушивание и принятие соединений.

1. **TUDPBlockSocket**:

Назначение: реализация UDP-соединений.

* Поддерживаемые протоколы: IPv4, IPv6.
* Основные функции: отправка и получение датаграмм, поддержка широковещательной и многоадресной передачи.

1. **Специализированные классы**:

* Примеры: THTTPSend (HTTP-запросы), TSMTPSend (отправка почты), TFTPsend (передача файлов).
* Назначение: упрощение работы с протоколами прикладного уровня, реализованы в соответствующих модулях.

1. **TBlockSocket** (базовый класс):

Назначение: предоставление общей функциональности для TCP и UDP сокетов.

Поддерживает настройку параметров (таймауты, размер буферов, TTL) и обработку ошибок.

## Требования к конфигурации

### Для работы Synapse необходимо добавить соответствующие модули в секцию uses программы (например, blcksock для сокетов, tlntsend для Telnet/SSH).

### Для использования шифрования необходимо установить и настроить внешние библиотеки (OpenSSL, Cryptlib, LibSSH2).

Настройка сетевых параметров (таймауты, TTL для multicast) выполняется программно через свойства классов.

# Действия при возникновении ошибок и сбоев в работе ПК «FANSY»

## Определение типа нештатной ситуации

Под ошибкой в ПК «FANSY» понимается остановка работы функции с выводом сообщения об ошибке.

Критическими ошибками считаются случаи, при которых:

* нарушается логическая целостность базы данных или становится невозможен вход/выход из системы;
* невозможно добавить, изменить или удалить бизнес-объект, что приводит к остановке технологического процесса;
* нарушается корректный обмен данными (выгрузка/загрузка документов), если процедуры обмена разработаны поставщиком и описаны в документации;
* некорректно отображаются остатки по счетам при правильном вводе данных и соблюдении инструкций;
* невозможно получить регламентированную отчётность из состава дистрибутивных форм;
* невозможна корректная загрузка сделок и котировок с биржевых площадок.

При обнаружении любой ошибки (критической или некритической) администратор обязан зафиксировать ситуацию и провести первичную диагностику.

## Первичные действия при появлении сообщения об ошибке

При обнаружении нештатного поведения Системы

* Не закрывайте окно с сообщением об ошибке сразу — запишите или сделайте скриншот:
* полный текст сообщения (включая код ошибки, если указан);
* точное время возникновения (в случае если его нет в полном тексте ошибки);
* учётную запись пользователя (важен префикс и перечень рабочих мест);
* действие, которое выполнялось непосредственно перед ошибкой (какой документ, форма, операция).

Закройте окно с ошибкой после фиксации информации.

Запустите встроенную диагностику последней ошибки:

В меню приложения выберите: Система → Диагностика последней ошибки.

Если кнопка не работает (почтовая программа не поддерживает или заблокирована) — используйте «Копировать» и отправьте вручную.

Дополнительно зафиксируйте контекст:

Сделайте скриншот экрана в момент ошибки (клавиша Print Screen).

Вставьте скриншот в документ (например, в текстовый редактор или LibreOffice) и сохраните вместе с диагностикой.

# Действия при возникновении ошибок и сбоев в работе ПК «FANSY»

Обновления ПК «FANSY» предназначены для устранения ошибок, добавления функциональности, улучшения безопасности или соответствия новым нормативным требованиям. Установка обновлений обязательна для поддержания работоспособности и уровня доверия 6.

ПК «FANSY» работает в клиент-серверной архитектуре, поэтому обновления обычно состоят из серверной и клиентской частей. Обновления не затрагивающие функции безопасности устанавливаются без дополнительных испытаний; обновления, влияющие на безопасность, требуют проверки целостности (раздел 13 и 18).

## Получение обновлений

Поставщик уведомляет об обновлении по электронной почте с темами вида:

«v4.55a: обновление базы данных с версии 4.12c до 4.13a (PostgreSQL) (дата)» — для серверной части.

«v4.55a Fansy-DEPO: обновление с версии 4.12c до 4.13a (дата)» — для клиентской части (указывается приложение: Fansy-DEPO, Fansy-SPECTRE и т.д.).

Скачайте архивы с обновлениями на сайте www.fansy-service.ru в разделе «Личный кабинет» (вход по логину и паролю).

Проверьте, что обновление предназначено для вашей версии (проверьте в заголовке приложения или свойствах \*.fansy- исполнимого файла).

В «Личном кабинете» также доступны:

«Новости» с описанием изменений;

обновлённая документация (например, «Документация Fansy-DEPO (дата)»).

## Содержание архивов обновлений

Серверная часть (архив с указанием СУБД: PostgreSQL, РЭД База Данных и т.д.):

* Директория SCRIPTS: SQL-скрипты для изменения структуры базы данных (выполняются в нумерационном порядке).
* Директория BASES: Образ новой базы метаданных (для PostgreSQL — .sql, для РЭД База данных – fbk, полностью заменяет текущую META).

Клиентская часть (архив с указанием приложения):

* Исполняемые файлы (SPECTRE.fansy, DEPO.fansy и т.д.).
* Шаблоны отчётов в директории REPORT.
* Файлы в директории MACROS (xtr, xsd-схемы).
* Обновлённая документация в директории.
* Файлы системных сообщений (\*.mes) в директории MESSAGES или корне архива (для изменения бизнес-логики).

Если каких-то директорий/файлов нет — обновление этого компонента не требуется.

## Подготовка к установке

* 1. Убедитесь, что все пользователи вышли из приложения. Рекомендуется временно переименовать исполнимый \*.fansy - файл (например, SPECTRE\_old.fansy), чтобы предотвратить вход.
  2. Создайте резервные копии:
* Базы DATA и META (см. раздел 14.5).
* Текущих файлов приложения (исполняемые, библиотеки, REPORT, MACROS).

## Установка серверной части

Общее: Проследите за выходом пользователей. Создайте дампы баз и протестируйте их восстановление (убедитесь, что нет ошибок в листинге). В конце года (если требуется) секционируйте таблицы FS\_OD\_DOCS (по D\_DATE), FS\_OD\_VALUE\_RATES (по OFICDATE), FS\_OD\_WIRING (по WIRDATE), PROP\_GARBAGE (по OP\_DATE) для производительности.

1. Для РЭД База Данных (Firebird):

* Выполните SQL-скрипты из SCRIPTS в нумерационном порядке (утилитой isql или IBExpert под SYSDBA).
* Разверните новый образ META из BASES (zzz\_SP\_META.fbk) с помощью GBAK или IBExpert — замените текущую META.
* Замените библиотеки из IB\_UDF (ib\_Fansy.dll / ib\_Fansy\_3.dll). Для Linux — скомпилируйте файлы.
* Если есть MAKER — выполните скрипты только по указанию поставщика.
* При ошибке — восстановите из резервной копии и свяжитесь с поставщиком.

1. Для PostgreSQL:

* Подключитесь под SYSDBA с ролью повышенных привилегий.
* Выполните скрипты PG\_DATA\_XXX.SQL в нумерационном порядке (pgAdmin или psql, под именем схемы приложения; используйте SET SCHEMA 'имя\_схемы';).
* Первый скрипт проверяет текущую версию базы (должна соответствовать файлу).
* Не копируйте/вставляйте скрипты — используйте прямое выполнение файла, чтобы избежать искажения текста.

Разверните новый образ META из BASES (.sql / .fbk) — замените текущую META.

При ошибке — восстановите из копии и свяжитесь с поставщиком. Не повторяйте скрипт без восстановления.

## Установка клиентской части

1. Сохраните старые файлы в архивную директорию.
2. Скопируйте из архива:

* Новые \*.fansy в директорию приложения (раздел 4.1 Руководства по установке и настройке (RU-62.01.29-000-52775472-2024-01 85)).
* Шаблоны в REPORT.
* Файлы в MACROS.
* Документацию в DOC.

1. Обработайте файлы \*.mes (если есть, в MESSAGES или корне):

* Скопируйте в специальную директорию (например, /home/fansy/POST или аналогичную, настроенную в системе).
* Запустите приложение под учтеной записью SYSDBA.
* Откройте Журнал документов (Меню → Форма → Открыть → Журнал документов).
* Выберите категорию «Прием сообщений из директории».
* Найдите/создайте документ с указанной директорией (в карточке).
* Выполните директиву «Прием файлов из почтовой директории и обработка полученных сообщений» (Меню → Документ → Выполнить).
* Если новый документ — сначала выполните «Регистрация».
